# Site Builder

The Site Builder lets you build a complete frontend website within Total CMS. Templates live inside the CMS, and disposable PHP stubs are generated to your docroot. Users who don't need the builder can ignore it entirely — it adds nothing to their existing workflow.

## How It Works

The builder follows a simple mental model:

1. **Templates** live in `tcms-data/builder/` — these are what you edit
2. **Page metadata** lives in a `builder-pages` collection — path, title, layout, status, sort order
3. **Stubs** are generated to your docroot — thin PHP files that boot T3 and render a template
4. **Generate is idempotent** — run it as many times as you want, same result

Each generated stub is intentionally minimal:

```php
<?php
// Generated by Total CMS Site Builder — do not edit
// Template: tcms-data/builder/pages/about.twig
require_once __DIR__ . '/../tcms-boot.php';
tcms_render('pages/about.twig');
```

The stub requires a shared `tcms-boot.php` file (also generated) that bootstraps Total CMS and provides the `tcms_render()` function. All rendering logic lives in your Twig templates — stubs are disposable and can be regenerated at any time.

## Template Primitives

Templates are organized into four directories inside `tcms-data/builder/`:

### Layouts (`layouts/`)

Base HTML structure. Pages extend layouts via Twig's `{% extends %}`. A site might have a `default.twig`, a `blog.twig`, and a `landing.twig`.

```twig
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>{% block title %}{{ cms.config('domain') }}{% endblock %}</title>
</head>
<body>
    {% include 'partials/nav.twig' %}
    <main>{% block content %}{% endblock %}</main>
    {% include 'partials/footer.twig' %}
</body>
</html>
```

### Pages (`pages/`)

The actual page content. Each page extends a layout and uses `cms.*` calls to pull content from T3 collections.

```twig
{% extends 'layouts/default.twig' %}

{% block title %}About — {{ cms.config('domain') }}{% endblock %}

{% block content %}
<section>
    <h1>About Us</h1>
    <p>Our story begins here...</p>
</section>
{% endblock %}
```

### Partials (`partials/`)

Reusable fragments included across pages and layouts — navigation, footer, sidebar, cards. Included via `{% include %}`.

```twig
{# partials/nav.twig #}
<nav>
    {% set pages = cms.collection.objects('pages', {sort: 'sort:asc'}) %}
    {% for page in pages %}
        {% if page.status == 'published' %}
        <a href="{{ page.path == '' ? '/' : '/' ~ page.path }}">{{ page.title }}</a>
        {% endif %}
    {% endfor %}
</nav>
```

### Macros (`macros/`)

Reusable Twig functions for repeated rendering patterns — e.g., a card component used inside collection loops.

## Pages Collection

Page metadata is stored in a standard T3 collection using the built-in `builder-page` schema. This means pages are queryable with all existing T3 tools (useful for nav menus, sitemaps, and admin management).

### Page Schema Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | slug | Unique page identifier (auto-generated from title) |
| `title` | text | Page title for `<title>` tag and navigation |
| `path` | text | URL path (e.g., `about` or `blog/archive`). Empty = homepage |
| `layout` | text | Layout template to extend (default: `default`) |
| `description` | textarea | Meta description for SEO |
| `status` | select | `published` or `draft`. Only published pages generate stubs |
| `sort` | number | Navigation ordering (lower numbers appear first) |
| `parent` | text | Parent page ID for hierarchical navigation |
| `updated` | datetime | Last modified (auto-set) |
| `created` | datetime | Creation date (auto-set) |

### Creating the Pages Collection

Create a new collection in the admin with:
- **Schema**: `builder-page`
- **Collection ID**: `builder-pages`

Or use `tcms builder:init` which creates it automatically.

## Path Mapping

The `path` field on a page object determines both the template location and the generated stub path:

| Page path | Template file | Generated stub |
|-----------|--------------|----------------|
| *(empty)* | `pages/index.twig` | `docroot/index.php` |
| `about` | `pages/about.twig` | `docroot/about/index.php` |
| `blog/archive` | `pages/blog/archive.twig` | `docroot/blog/archive/index.php` |
| `services` | `pages/services.twig` | `docroot/services/index.php` |

## Accessing the Builder

The Builder section appears in the admin sidebar for users with template access permissions. It replaces the previous Templates section and manages all template types — layouts, pages, partials, macros, and general-purpose templates.

To get started with site building, run `tcms builder:init` to scaffold a starter site, or create templates manually in the admin.

## Coexistence with Stacks

Stacks-published pages and T3-generated stubs coexist peacefully in the same docroot. Generated stubs only create files for pages defined in your `builder-pages` collection. A Stacks user on the same T3 install sees no change to their existing workflow.
