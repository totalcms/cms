# Git Setup for Total CMS Projects

This guide helps you set up version control for your Total CMS project, including which files to track and which to ignore.

## Quick Setup

1. Initialize git in your project directory:
   ```bash
   cd your-totalcms-project
   git init
   ```

2. Copy the `.gitignore` file below and save it as `.gitignore` in your project root

3. Add your files:
   ```bash
   git add .
   git commit -m "Initial Total CMS project setup"
   ```

## What Gets Preserved vs Ignored

### ✅ **Preserved (Tracked in Git)**
- **Your content**: All collections, objects, and data in `tcms-data/`
- **Your customizations**: Templates, schemas, configuration
- **Your code**: Any custom PHP, JavaScript, or CSS
- **Project files**: composer.json, package.json, documentation

### ❌ **Ignored (Not Tracked)**
- **Generated files**: Built assets, cache files, logs
- **Dependencies**: vendor/, node_modules/
- **Temporary files**: uploads, sessions, tmp files
- **Environment config**: Local settings, API keys
- **System files**: OS-specific files (.DS_Store, Thumbs.db)

### ⚠️ **Special Case: Watermarks**
The `/tcms-data/.watermarks/` directory contains generated watermark images that are automatically recreated. These are ignored to keep your repository clean.

## Managing Large Media Files

If your `tcms-data/` directory contains large media files, you have several options:

### Option 1: Include Everything (Default)
Keep all media files in version control. Good for:
- Small to medium projects
- Teams that need all assets
- Complete backup in git

### Option 2: Exclude Large Media
Uncomment the media file patterns in the `.gitignore`. Good for:
- Projects with large image/video collections
- Keeping repository size manageable
- Faster git operations

**Note**: If you exclude media files, you'll need to manage them separately (FTP, cloud storage, etc.)

### Option 3: Use Git LFS
For large files you want to track, consider [Git LFS](https://git-lfs.github.io/):

```bash
# Install Git LFS
git lfs install

# Track large file types
git lfs track "*.jpg"
git lfs track "*.png" 
git lfs track "*.mp4"
git lfs track "*.pdf"

# Add and commit
git add .gitattributes
git commit -m "Add Git LFS tracking"
```

## Common Workflows

### Initial Setup
```bash
# Clone or create your project
git init
# or
git clone your-repo-url

# Copy the .gitignore content above to .gitignore
# Add your files
git add .
git commit -m "Initial Total CMS setup"
```

### Regular Development
```bash
# Check what's changed
git status

# Add your changes (media files, content, code)
git add .
git commit -m "Update content and features"

# Push to remote
git push origin main
```

### Deploying to Production
```bash
# On your production server
git pull origin main

# Build assets (if needed)
composer install --no-dev --optimize-autoloader
npm install --production
npm run build

# Set proper permissions
chmod -R 755 tcms-data/
chmod -R 755 cache/
chmod -R 755 logs/
```

## Troubleshooting

### "File too large" errors
- Check if large media files are being tracked
- Consider using Git LFS or excluding media files
- Use `git lfs migrate` to move existing large files to LFS

### Missing content after clone
- Ensure media files aren't being ignored unintentionally
- Check that `tcms-data/` patterns aren't too broad
- Verify team members have the same `.gitignore`

### Permission issues
- Ensure web server can write to `cache/`, `logs/`, and `tmp/`
- Check that `tcms-data/.watermarks/` can be created and written to

### Cache not regenerating
- The `.watermarks/` directory is ignored, so it won't be in your repository
- Total CMS will automatically recreate watermarks when needed
- Make sure the web server has write permissions to `tcms-data/`

## Best Practices

1. **Review your .gitignore** - Make sure it matches your project needs
2. **Test after cloning** - Verify your site works after a fresh git clone
3. **Document your setup** - Note any special requirements for your team
4. **Back up separately** - Consider additional backups beyond git
5. **Monitor repository size** - Keep an eye on how large your repo gets
6. **Use branches** - Create feature branches for major changes
7. **Regular commits** - Commit content and code changes regularly
