## Project Brief: Site Builder

**Goal**
Give T3 an optional, complete frontend website building system. Users who want to build their entire site within T3 can do so without needing Stacks, Pinegrow, or any other external tool. Users who don't need it ignore it entirely — it adds nothing to their existing workflow.

**Constraints**
- Ships as a first-party extension (built on the Extensions architecture)
- Generated PHP stubs are disposable — templates inside T3 are always the source of truth
- No visual drag-and-drop editor — file-based workflow with a code editor in the admin
- No opinion on JS/CSS build pipelines — T3 recommends Vite but doesn't own it
- Stacks-generated pages and T3-generated stubs coexist peacefully in docroot
- Borrows heavily from Proton — do not reinvent what already works there

---

### Architecture: Generate PHP Stubs to Docroot

Exactly as defined in your existing plan. The key mental model is:

- **Templates live inside T3** (`tcms-data/templates/`) — this is what you edit
- **Stubs live in docroot** — these are generated, never hand-edited
- **Generate is idempotent** — run it as many times as you want, same result

Each generated stub is intentionally thin:

```php
<?php
// Generated by Total CMS — do not edit this file
// Edit the template at: tcms-data/templates/pages/about.twig
require_once '/path/to/tcms/tcms-boot.php';
tcms_render('pages/about.twig');
```

---

### Template Primitives

Four primitives, borrowed directly from Proton:

**Layouts** (`tcms-data/templates/layouts/`)
Base HTML structure. Pages extend layouts via Twig's `{% extends %}`. A site might have a `default.twig`, a `blog.twig`, and a `landing.twig`.

**Pages** (`tcms-data/templates/pages/`)
The actual page content. Each page maps to a URL path and extends a layout. Contains `cms.*` calls to pull content from T3 collections.

**Partials** (`tcms-data/templates/partials/`)
Reusable fragments included across pages and layouts — nav, footer, sidebar, cards. Included via `{% include %}`.

**Macros** (`tcms-data/templates/macros/`)
Reusable Twig functions for repeated rendering patterns — e.g. a `render_card(item)` macro used inside collection loops.

Assets are explicitly out of scope here — T3 already handles media. Static assets (CSS, JS) live in docroot and are managed by the developer's build tool of choice.

---

### Admin UI: Site Structure Section

A new admin section (registered as an extension) with four areas:

**Template Editor**
- File tree of `tcms-data/templates/` on the left
- Monaco editor (or CodeMirror) in the center with Twig syntax highlighting
- Preview pane on the right — renders the current template against live T3 data
- Save writes directly to the template file

**Page Tree**
- Visual list of all pages with their URL paths, assigned layouts, and assigned templates
- Add / edit / delete pages
- Each page entry: path, template, layout override, meta fields (title, description)
- "Generate" button writes all stubs to configured docroot

**Layout Manager**
- List of available layouts
- Create / edit / delete layouts
- Each layout shows which pages are using it

**Settings**
- Docroot path configuration
- Base URL
- Auto-generate on save toggle
- Vite integration config (output directory for compiled assets)

### CLI Commands

```bash
# Generate all page stubs to docroot
tcms generate --output /var/www/html

# Generate a single page
tcms generate --page about --output /var/www/html

# Preview what would be generated without writing
tcms generate --dry-run

# Watch mode — regenerate stubs when site structure changes
tcms watch

# Scaffold a new site from a starter template
tcms init --starter=business
tcms init --starter=blog
tcms init --starter=portfolio
```

---

### Starter Templates

The `tcms init` starters are the onboarding accelerator. Each one provides a complete working site structure out of the box:

- **business** — home, about, services, contact, blog
- **blog** — home, archive, single post, category pages
- **portfolio** — home, work index, case study, about, contact
- **docs** — home, section index, doc page, search

Each starter includes:
- Pre-built layouts, pages, partials
- Matching T3 schemas for its content types
- A `README.twig` in the template root explaining the structure
- A JumpStart file with factory-generated demo content

This is where `tcms push` becomes valuable — a developer can `tcms init --starter=business` locally, customize it, then `tcms push` to production when ready.

---

### JS/CSS Build Pipeline

T3 does not own this but should have a recommended path. The starter templates each include a `vite.config.js` that:

- Takes source files from `tcms-data/assets/src/`
- Outputs compiled files to `docroot/assets/`
- Runs `npm run dev` for local development with hot reload
- Runs `npm run build` for production

T3 documentation covers this setup clearly. BSH could offer a node environment on hosting plans as a differentiator for users who want the full pipeline on the server.

---

### Borrowing From Proton

Per your existing plan, take directly from Proton:

- ✅ Layout rules system (route-based layout assignment)
- ✅ Init/scaffolding CLI
- ✅ Watch mode (pure PHP, no BrowserSync)
- ✅ Partials/macros organization conventions
- ✅ Front matter for page-level config

Do not take:
- ❌ Static HTML compilation (pages are live, not compiled)
- ❌ YAML data layer (T3 collections are the data source)
- ❌ HTML minification
- ❌ Laravel Zero framework

### What Done Looks Like

- `tcms init --starter=business` scaffolds a complete working site structure in under 60 seconds
- Template editor in admin opens, edits, and saves Twig files with syntax highlighting
- Preview pane renders a template against live T3 data
- Page tree shows all pages, allows adding a new page, assigning a template and layout
- `tcms generate` writes correct PHP stubs to docroot
- Generated stubs render correctly when visited in a browser
- Stacks-published pages and T3-generated stubs coexist in the same docroot without conflict
- `tcms watch` detects structure changes and regenerates stubs automatically
- A Stacks user on the same T3 install sees no change to their existing workflow
