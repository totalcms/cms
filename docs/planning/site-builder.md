# T3 Site Builder — Feature Plan

**Status:** Planning phase (2026-03-13)
**Related project:** ~/Developer/proton (Proton static site builder, PHP/Twig/Laravel Zero)

## Vision

T3 gets a built-in site structure generator — a bundled tool that helps users build their website's page structure. It is **optional** (T3 still works with any frontend: Stacks, custom, etc.) but provides a complete solution for users who need one.

This is NOT a static site generator. The generated pages are live — they bootstrap T3's Twig engine and pull content from T3 at runtime.

## Architecture: "Generate PHP stubs to docroot"

### How It Works

1. **User designs in T3 admin** — creates/edits layouts, pages, partials in a template editor (Twig files stored in `tcms-data/templates/`)
2. **User defines site structure** — a page tree: `/`, `/about`, `/blog`, `/contact`, each pointing to a template and a layout
3. **User clicks "Generate Site"** (or runs `tcms generate`) — T3 writes thin PHP stubs to docroot
4. **Site is live** — Apache/nginx serves the PHP files, each one bootstraps T3 and renders the assigned template

### What Gets Generated (to docroot)

Thin PHP entry point stubs only:

```php
<?php
// Generated by Total CMS — do not edit this file
// Edit the template in Total CMS admin instead
require_once '/path/to/totalcms/tcms-boot.php';
tcms_render('pages/about.twig');
```

### What Stays Inside T3 (`tcms-data/templates/`)

- Layouts (layout inheritance, `{% extends %}`)
- Page templates (the actual Twig with `cms.*` calls)
- Partials (`{% include %}`)
- Macros
- Site structure configuration

### The `tcms-boot.php` Bootstrap

A lightweight bootstrap that:

- Loads T3's autoloader and DI container
- Sets up the Twig environment with all `cms.*` adapters
- Provides `tcms_render()` function
- Reuses existing `TwigEngine` and adapter classes
- Skips admin routes and middleware (fast path)

## CLI Tool: `tcms`

```bash
# Generate all pages to docroot
tcms generate --output /var/www/html

# Generate a single page
tcms generate --page about --output /var/www/html

# Preview what would be generated
tcms generate --dry-run

# Watch mode (regenerate stubs when structure changes)
tcms watch

# Init a starter site structure
tcms init --starter=business
```

## Admin UI

A "Site Structure" section in T3 admin:

- **Page tree** — visual list of pages with URL paths and assigned templates
- **Layout manager** — manage layout files
- **Partials library** — reusable template components
- **Generate button** — writes PHP stubs to configured docroot
- **Settings** — docroot path, base URL, auto-generate on save toggle

## Compatibility

- **Stacks users:** Stacks-generated pages coexist alongside T3-generated stubs in docroot. Users can mix both.
- **Custom frontend users:** Ignore this feature entirely. Build their own PHP/HTML as before.
- **Pure T3 users:** Get a complete site builder without needing another tool.
- **Generated files are disposable** — regenerate anytime. Templates inside T3 are the source of truth.

## What's Borrowed From Proton

- Layout rules system (route-based layout assignment, e.g. `blog/*` → `blog-layout.twig`)
- Init/scaffolding CLI (generate site structures from starter templates)
- Watch mode (100% PHP, no BrowserSync — Proton already rewrote this)
- Partials/macros organization conventions
- Front matter for page-level config (layout override, meta, etc.)

## What's NOT Borrowed From Proton

- Static HTML compilation (pages are live, not compiled)
- YAML data layer (T3 collections are the data source)
- Batch page generation (T3's Load More handles this)
- Sitemap generation (T3 already has this)
- HTML minification/pretty-printing
- Laravel Zero framework (T3 CLI would use its own patterns)

## Open Questions

- Data model for site structure config (how are pages/layouts/routes defined inside T3?)
- Starter template designs (what site types to offer: blog, portfolio, business, docs?)
- Should watch mode auto-generate stubs, or just live-reload template changes?
- How does the page tree interact with T3's existing template system and Template Designer?
- Pricing: bundled free (current thinking) vs. paid addon via future extensions API?

## Future Consideration: Extensions API

This feature was discussed in the context of a future T3 extensions API. Current thinking is this should be a **free bundled feature** (it makes T3 more approachable), and paid addons should be reserved for things that add ongoing value (e-commerce, analytics, etc.). The extensions API is a separate initiative.
